1. CORS починен — allow_credentials=False при allow_origins=["*"]. Убрана невалидная комбинация, из-за которой браузер блокировал бы запросы внешнего фронтенда. Preflight проверен:
   access-control-allow-origin: \* без credentials.

2. Единый контракт в обоих режимах — mock /api/query приведён к RAG-форме (to_rag_api()/mock_search_api() в app/query_engine.py). Проверено по HTTP: ключи ответа идентичны для rag и mock:

- top-level: answer, contradictions, evidence, gaps, graph, mode, parsedQuery, queryId, sources ✓
- evidence: id, text, score, confidence, conditions, matchedTerms, numericStatus, source ✓
- answer, source, graph.edges — 1:1 ✓

Фронтендер пишет код под одну форму, независимо от наличия индекса.

3. docs/API_APP.md — полная спецификация для фронтендера: раздел «как поднять фронт против бэка» (base URL, CORS, fetch-примеры), контракт SearchResult с JSON-примерами, все эндпоинты
   (query/graph/sources/dashboard/export/upload/auth/…), формат ошибок, чек-лист интеграции. Связана из README, ссылается на rag/docs/API.md.

4. run.py с аргументами — --host / --port / --reload / --db (+ env HOST/PORT/RAG_DB_PATH). Путь к индексу связан с мостом через RAG_DB_PATH — индекс можно держать вне проекта.

Итог для фронтенд-разработчика

Достаточно docs/API_APP.md. Он может:

- разрабатывать без 5.8 ГБ индекса (demo-режим, тот же контракт);
- запускать бэк на любом порту (py run.py --port 8000), фронт — на своём;
- строить UI вокруг стабильного SearchResult, встроенный static/index.html — живой референс вызовов.

Одно замечание: встроенный демо-UI и внешний фронтенд не конфликтуют (внешний просто дёргает API). Если ваш фронтендер захочет — встроенный UI можно оставить как «эталон поведения» или отключить (отдаётся
только на /).
